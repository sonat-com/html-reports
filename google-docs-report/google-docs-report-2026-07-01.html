Google Docs Feature — Usage & Drop-off.

Report date: July 1, 2026  ·  Window: All available history  ·  Data range: Jan 30 – Jul 1, 2026 (~5 months)  ·  Funnel window: 14 days

Executive Summary

The Google Docs feature has seen 13 unique users across all four flows since instrumentation began (January 30, 2026). The Editor "Create new Google Doc" path is the most-used entry point (8 users), followed by Import (4 users) and the Google Docs Add-on (2 users, dev/test only in February 2026). The Embed flow has no telemetry.

End-to-end completion is extremely low: of 11 users who expressed Google Doc intent, only 1 ever published a Google Doc (9.1% intent-to-completion). That single user published 12 times in a 4-day burst (May 20–24), suggesting internal or power-user testing rather than broad adoption.

Two instrumentation gaps obscure the true picture: (1) the is_google_doc property on document_created is never set to true, making the Editor funnels appear to have 0% conversion at step 2 — a false signal; and (2) the "Reconnect" action on embed disconnect fires no event. A frustration signal is also visible: 2 of 11 intent users (18.2%) rage-clicked during their session.

Instrumentation Gaps ACTION REQUIRED

  • 1. is_google_doc on document_created — never set to true. Both Editor funnels (Create and Import) return 0 users at step 2. This makes the funnels useless until fixed.
  • 2. Embed "Reconnect" action — when an embedded Google Doc loses its connection, the user must reconnect. No event fires for this interaction. Connection-loss frequency and reconnect success rate are invisible.
  • 3. addon_connect_success — this event does not exist in PostHog. The real event fired by the add-on is addon_doc_connected. Weekly trend and retention queries that reference addon_connect_success return all zeros. Retention data is therefore entirely empty.

Friction Signal

10 rage clicks from 2 users (18.2% of 11 intent users) indicate meaningful UI friction in the Google Doc flows. Without session recordings, the specific breakpoint is unknown.

Strengths

  • Add-on sync is perfectly reliable: 7 of 7 sync operations succeeded (100%) in the available test data.
  • Add-on publish reliability: 2 of 3 publish operations succeeded (66.7%). The failure was a 404, suggesting an edge case rather than a systemic bug.
  • Once connected, the add-on flow completes: After the account-selection cliff, the connect → connect_success → publish_initiated → publish_success path runs at 100% per-step conversion.
  • One power user validates the end-to-end flow: 12 Google Doc publishes from a single user in 4 days (May 20–24) confirms the full Editor → Publish path works end-to-end.
  • Editor intent is picking up slowly: The May 17, 2026 week was the peak (4 create + 4 import clicks from 8 intent-events), suggesting growing awareness of the feature.

Recommendations

1. Fix is_google_doc on document_created (highest priority)
Observation: Both Editor funnels show 0% conversion at step 2 because the property is never set. Hypothesis: a single code-path fix in the Editor will restore the funnel and reveal the true Create and Import conversion rates. Expected impact: recovers visibility into the most-used entry paths.
2. Investigate the 10-of-11 intent users who never published
Observation: 90.9% of users who clicked a Google Doc action never completed a publish. Hypothesis: the Google OAuth / Picker handshake is failing silently, or users are confused after the picker closes. Expected impact: fixing the most common failure point could multiply completion rates significantly from the current 9.1%.
3. Add rage-click session recordings / heatmaps for Google Doc intent users
Observation: 18.2% of intent users rage-clicked. Hypothesis: a specific button or loading state is unresponsive (likely during OAuth popup or Picker). Expected impact: pinpoints the exact friction point without guesswork, enabling a targeted fix.
4. Instrument the Embed "Reconnect" action
Observation: No events fire when a disconnected embedded Google Doc reconnects. Hypothesis: connection loss may be frequent in production but invisible. Recommended events: embed_reconnect_initiated, embed_reconnect_success, embed_reconnect_failed. Expected impact: makes connection reliability measurable.
5. Align addon_connect_success event name (or update queries)
Observation: PostHog receives addon_doc_connected, not addon_connect_success. Retention and weekly trend queries referencing the latter return all zeros. Hypothesis: a naming mismatch between the add-on code and the analytics schema. Fix: either fire addon_connect_success as an alias or update all PostHog queries to use addon_doc_connected.
6. Investigate the 404 on addon_connect_failed
Observation: 2 connect failures returned HTTP 404 with "Unknown error". Hypothesis: the add-on's connect API endpoint is missing or mis-routed for a specific account/doc configuration. Expected impact: unblocking this removes the only known hard error in the add-on connect flow.
