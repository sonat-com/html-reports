Google Docs Feature — Usage & Drop-off

Window: all available history · generated Jul 1, 2026

True data history: Jan 30 – Jul 1, 2026 (~5 months). Add-on telemetry: Feb 24–27, 2026 only (3-day dev/test window). Google Doc publishes: May 20–24, 2026.

Executive summary

Across all history, 13 unique users touched at least one of the four Google Docs flows. The dominant entry point is the Editor: 8 users clicked “Create new Google Doc” and 4 clicked “Import Google Doc”; the Google Docs Add-on drew just 2 users.

The defining problem is completion. Of the 11 users who expressed intent in the Editor, only 1 (9.1%) ever published a Google Doc — and that single user produced all 12 Google Doc publishes on record, over a 4-day burst in May. 10 of 11 intent users (90.9%) produced no Google Doc at all. That intent→completion gap is the biggest cliff in the entire feature.

The Add-on’s end-to-end path (sign-in → publish) is clean but empty: 2 test users, a single 50% drop at account selection, and otherwise 100% step-through — plus 7/7 syncs and 2 of 3 publishes succeeding with zero sync/publish failures. Reliability is not the issue; reach is.

Add-on framing: No evidence the add-on is published: this run could not confirm a public Marketplace listing, and add-on usage is limited to 2 dev/test account(s) with no external business-domain adoption. Treat the add-on funnel as dev/test-only, not a conversion failure. (If the add-on has since been published, external-domain usage will appear and this note clears automatically.)

Editor funnels — Google Doc intent

The Editor funnels’ “created / imported / published” steps read zero because they key on the is_google_doc=true property flag, which the app never sets — an instrumentation artifact, not a literal 100% failure. The trustworthy completion signal is document_published with content_type=GoogleDoc: 12 publishes by 1 user. See the intent→completion breakdown below.

Biggest funnel cliffs

  • Intent → completion: −10 users (−90.9%) — 10 of 11 intent users never publish a Google Doc. Top priority
  • Add-on Signed in → Account selected: −1 user (−50%) — the only add-on drop (dev/test scale).
  • Editor create/import step-2 reads −8 / −4 but is an instrumentation artifact (broken is_google_doc flag), not a true cliff.

Known instrumentation gaps blind spots

  • Editor “Reconnect” (disconnect-recovery) fires no event. How often embedded/connected Google Docs lose their connection — and how often users recover — is completely invisible. Recommend instrumenting it.
  • is_google_doc=true is never set on document_created. The create/import completion steps can’t be measured through that flag; completion is only observable via document_published (content_type=GoogleDoc).
  • addon_connect_success / addon_opened are never emitted (the app uses addon_doc_connected). This zeroes out the weekly add-on-connect trend and makes add-on connect-retention uncomputable.

Strengths

  • Sync is rock-solid: 7 of 7 add-on syncs succeeded, 0 failures (100%).
  • Publishing works when reached: 2 of 2 recorded publish outcomes succeeded (0 publish failures); 3 initiated.
  • The add-on happy path is smooth: once past account selection, every remaining step converted 100% (connect → doc connected → publish initiated → published).
  • Deep engagement once activated: the single activated Google Doc user published 12 docs in 4 days — evidence the payoff is compelling for those who cross the completion line.
  • Low-friction entry: the Editor entry points reliably capture intent (11 distinct users, no failures at the click step).

Recommendations

1. Close the intent→completion cliff (top priority)

Observation: 10 of 11 intent users (90.9%) click into a Google Doc flow but never publish one; all 12 publishes come from a single user.

Hypothesis: the path from clicking “New/Import Google Doc” to a published Sonat topic has friction or an unclear next step (possibly OAuth/picker abandonment or an unfinished editing loop).

Expected impact: even lifting completion from 9.1% to 30% would ~3× the number of activated users and is the single highest-leverage move available.

2. Fix the is_google_doc flag so completion is measurable

Observation: document_created never carries is_google_doc=true, so the create/import funnels read zero and the true created-doc step is invisible.

Hypothesis: a one-line property fix restores an accurate created→published funnel and lets us localize where in rec. #1 users actually drop.

Expected impact: converts the biggest cliff from “unmeasurable” to diagnosable; prerequisite for iterating on rec. #1.

3. Instrument the Editor “Reconnect” action

Observation: disconnect-recovery for embedded/connected Google Docs emits no event; connection-loss frequency is a total blind spot.

Hypothesis: silent reconnection pain is suppressing repeat usage of embeds — we simply can’t see it today.

Expected impact: quantifies a suspected reliability drag and unblocks a data-backed fix; cheap to add.

4. Emit the real add-on success events (addon_connect_success)

Observation: the app fires addon_doc_connected but not addon_connect_success, zeroing the weekly connect trend and making connect-retention uncomputable; 10 of 13 connect initiations have no recorded terminal event.

Hypothesis: aligning event names to the funnel/retention queries restores connect success-rate and stickiness measurement.

Expected impact: unlocks add-on connect success-rate and retention — essential the moment the add-on is published.

5. Resolve the connect 404 (“Unknown error”)

Observation: the only recorded failure is addon_connect_failed — 404, “Unknown error”, 2 occurrences (1 user).

Hypothesis: a missing/renamed endpoint or resource returns 404 during connect; the generic message hides the root cause.

Expected impact: removes a hard failure on the most-clicked add-on action (connect) before wider release.

6. Re-activate a dormant feature

Observation: Google Doc intent peaked the week of May 17 (8 clicks) and has been zero for the last three weeks.

Hypothesis: with no in-product surfacing or announcement, the flows are simply not being discovered.

Expected impact: a targeted nudge to Free/Enterprise users who’ve created docs could restore and grow weekly intent from its current floor of zero.

Methodology & caveats

Source. Every figure comes from PostHog (project 300407), fetched by tools/fetch_posthog_data.py into output/posthog-data.json. There are no backend events for this feature — PostHog is the sole source for usage. No numbers are estimated; empty sections are reported as empty.

Window & data age. Requested window: all available history. True instrumented history runs Jan 30 – Jul 1, 2026 (~5 months) for Editor intent, but the addon_* events exist only for a 3-day dev/test session (Feb 24–27, 2026), and every Google Doc publish falls in a single 4-day span (May 20–24, 2026). Add-on data is therefore thin and dev/test-only — read those figures as directional, not adoption.

Funnel window. Users had 14 days to complete each funnel from its first step. Event inventory and the two heaviest cross-join queries were bounded to a 730-day lookback for query performance; this fully covers the feature’s instrumented life (earliest event Jan 30, 2026).

Add-on publish status (this run). Determined by tools/check_addon_published.py → output/addon-publish.json. Verdict: not_published, confidence medium, basis external_adoption_absent — no public Marketplace listing could be confirmed and add-on usage is 2 accounts, both consumer/personal domains, 0 external business-domain accounts. The add-on funnel is framed as dev/test throughout; the verdict self-corrects on a future run if the add-on ships.

Events vs users. “Events” are total occurrences; “users” are unique distinct_id. Note that raw document_published (2,126 events / 63 users) and create_button_clicked (531 / 152) span all content types — Google-Doc-specific figures use the action-scoped and content_type=GoogleDoc queries only.

Errors this run. None — all data sections returned successfully (_errors empty after query tuning).

Privacy. Aggregate only — counts, rates and segments (by plan / domain class). No emails, names, or user-id lists appear in this report.
