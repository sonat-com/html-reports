Executive summary

Web (in-Editor) — the real product story. 11 unique users engaged the in-Editor Google-Doc flow (8 started a New Google Doc, 4 an Import; 12 New clicks and 8 Import clicks in all). But only 1 of them ever published a Google Doc — a 90.9% intent→completion drop-off — and that single publisher produced all 12 publishes in one 4-day burst (May 20–24, 2026). Real adoption of the feature has effectively not materialised yet: engagement is a trickle, and completion is one power user.

Add-on (Marketplace sidebar). No evidence the add-on is published: this run could not confirm a public Marketplace listing, and add-on usage is limited to 2 dev/test account(s) with no external business-domain adoption. Treat the add-on funnel as dev/test-only, not a conversion failure. (If the add-on has since been published, external-domain usage will appear and this note clears automatically.)

Instrumentation & data caveats

  • Editor “Reconnect” (disconnect-recovery) fires no event, so connection-loss frequency is invisible — a blind spot worth closing.
  • Recently closed (Editor-Frontend PR #307): importing an existing Google Doc used to fire no completion event, and “Open in Google Doc” was untracked. Both now emit events, so their history starts at PR #307’s deploy — older data is empty by construction, not because the actions are unused.
  • No backend events exist for this feature — all usage is client-side PostHog, so measurement is bounded by what the frontend emits.

Recommendations

  1. Web Close the intent→publish cliff. Observation: 10 of 11 intent users never published a Google Doc, and all 12 publishes came from one user. Hypothesis: there is a hard blocker between clicking “New/Import Google Doc” and a published doc (auth handoff, doc creation, or an unclear next step). Expected impact: converting even 30% of intent users to a first publish would roughly triple the count of active publishers.
  2. Web Deploy & verify PR #307 completion instrumentation. Observation: “Created/Imported OK” and “Open in Google Doc” show no data and no first-seen — newly instrumented, not yet collecting. Hypothesis: without the created/imported completion event the entire middle of both funnels is a blind spot, so we can’t tell whether users fail to create or just aren’t tracked. Expected impact: makes the create/import drop-off diagnosable instead of inferred.
  3. Web Instrument the “Reconnect” action. Observation: disconnect-recovery fires no event; the 10 rageclicks among 2 intent users hint at silent friction. Hypothesis: unmeasured reconnect/disconnect churn is a hidden contributor to the intent→publish drop. Expected impact: quantifies a suspected friction driver so it can be prioritised.
  4. Add-on Publish the add-on to the Marketplace. Observation: Step 2b verdict is not published; usage is 2 dev/test accounts with zero external business-domain adoption. Hypothesis: the add-on cannot gain adoption until it is publicly listed — the “fix” is publishing, not funnel tuning. Expected impact: unlocks the only path to real add-on adoption and produces a real funnel to measure.
  5. Add-on Fix the connect 404 before launch. Observation: addon_connect_failed = “Unknown error”, status 404, and no addon_connect_success events against 13 connect initiations. Hypothesis: a broken connect endpoint would block adoption on day one of a public launch. Expected impact: prevents a launch-day connect failure that would waste acquisition.

Methodology

  • Source: PostHog only (project 300407) — no backend events exist for this feature. Every figure is from output/posthog-data.json; empty sections mean the event is unused or uninstrumented, never an estimate.
  • Window: all available history. True data history: Jan 30, 2026 – Jul 2, 2026.
  • Funnel window: 14 days for a user to complete a funnel from its first step.
  • Per-track user counts (never blended): Web = intent_to_completion.intent_users (11); Add-on = users_per_flow.addon_users (2). The blended total_unique_users is deliberately not featured.
  • Add-on publish status was determined at run time (output/addon-publish.json): verdict not_published, basis external_adoption_absent, confidence medium. When the add-on ships, external-domain usage appears and this framing clears automatically.
  • Query errors this run: None — all queries returned.
  • Aggregate only — no PII. Counts, rates, and plan segments only; no emails, names, or user-id lists.
