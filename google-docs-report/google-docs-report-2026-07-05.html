Executive summary

The Google Docs feature is three separate products with different audiences and maturity. They are reported in walled-off tracks and never blended — no combined user total, funnel, or rate spans more than one.

Web · in-Editor  This is where real product adoption lives. 11 unique users showed intent (clicked Create or Import a Google Doc), but only 1 ever published a Google Doc — a 9.1% intent→completion rate. That one publisher accounts for all 12 Google-Doc publishes; the other 10 intent users produced none. The single biggest reliably-measured cliff is intent → published (11 → 1 users).

Connect add-on  No evidence the add-on is published: this run could not confirm a public Marketplace listing, and add-on usage is limited to 2 dev/test account(s) with no external business-domain adoption. Treat the add-on funnel as dev/test-only, not a conversion failure. (If the add-on has since been published, external-domain usage will appear and this note clears automatically.)

Video Embed add-on  The Video Embed add-on is published to the Google Workspace Marketplace (verified: public listing returns HTTP 200 as of 2026-07-05). However, the Marketplace lists ~82 installs while PostHog records ZERO video-embed events — the deployed build predates the PostHog instrumentation and was never redeployed. This is an instrumentation gap, NOT low adoption: redeploy the instrumented build (Google-Workspace-Add-ons) to begin collecting usage.

Instrumentation & data caveats

  • Editor "Reconnect" is untracked. The disconnect-recovery action fires no event, so editor connection-loss frequency is invisible. Recommend adding it.
  • Recently closed (Editor-Frontend PR #307). Importing an existing Google Doc previously fired no completion event, and the Open in Google Doc button was untracked. Both now emit events, so their history starts at PR #307's deploy — older data is empty by construction, not because they are unused.
  • Web "embed a Google Doc as topic content" emits no telemetry and cannot be measured; stated once, not estimated.
  • Video Embed published-but-silent. Track C is live (~82 installs) yet records zero events — an instrumentation gap flagged above, not zero adoption.

Recommendations

Web

Close the intent→publish completion cliff

Observation
11 users show clear intent (click Create/Import), but only 1 ever publishes a Google Doc (9.1%); the other 10 produce nothing, and 10 rageclicks cluster among 2 of them.
Hypothesis
Friction between doc creation and first publish (or an early blocker in the create/import handoff) stalls users before their first published doc.
Expected impact
Lifting completion from 9.1% to even ~30% would roughly 3× the number of real users who publish a Google Doc.
Web

Deploy the PR #307 completion instrumentation so the create funnel becomes measurable

Observation
Both editor funnels show a 100% click→created "cliff" that is actually the newly-instrumented document_created (Google-Doc) row and the Open in Google Doc row reading zero with no first-seen date.
Hypothesis
Docs are being created, but the completion event isn't emitting on the currently-deployed editor build.
Expected impact
Deploying/verifying PR #307 removes a false 100% drop from the dashboard and exposes the true create→publish funnel, so the completion fix above can be aimed precisely.
Video Embed

Redeploy the instrumented Video-Embed build to stop the feature being invisible

Observation
The published sonat_videoembed listing reports ~82 installs but emits zero addon_video_insert_* events.
Hypothesis
The deployed Apps Script build predates the Feb-2026 PostHog instrumentation and was never redeployed.
Expected impact
A redeploy turns a live-but-silent feature into a measurable one — you'd finally see whether those ~82 installs convert into video inserts, and the YouTube/Vimeo split.
Connect

Publish the add-on (and fix the 404 connect failure first) — don't tune an unshipped funnel

Observation
The Connect add-on has 2 dev/test users, a single Feb-2026 burst, no external adoption, and every connect attempt failed with HTTP 404; it is not published to the Marketplace.
Hypothesis
It cannot drive adoption while unlisted, and the 404 connect error would block real users on day one.
Expected impact
Fixing the connect 404 and publishing the listing is what unlocks a real external funnel — a conversion-optimization pass is premature until then.
Web

Instrument the editor "Reconnect" action

Observation
Disconnect-recovery fires no event, so how often editor connections drop — and how often users recover — is completely invisible.
Hypothesis
Silent reconnects may be a hidden friction source contributing to the low completion rate.
Expected impact
Adding the event quantifies a currently-invisible failure mode and lets connection loss be ruled in or out as a completion blocker.

Methodology

  • Source: PostHog only (project 300407) — there are no backend events for this feature. Every figure is read from output/posthog-data.json; empty sections mean the event is unused or uninstrumented, reported as-is, never estimated.
  • Window: all available history (-3650d). True data history: Jan 30, 2026 – Jul 3, 2026. Funnel window: 14 days.
  • Track separation: Web users = intent_to_completion.intent_users (11); Connect users = users_per_flow.addon_users (2); Video-Embed users = users_per_flow.video_embed_users (0). No blended total is headlined; no metric, rate, or chart spans more than one track.
  • Connect publish status (output/addon-publish.json): verdict not_published · basis external_adoption_absent · confidence medium.
  • Video-Embed publish status (output/videoembed-publish.json): verdict published · basis marketplace_listing · confidence high · Marketplace installs ~82 vs 0 PostHog events.
  • Data errors this run: none.
  • Privacy: aggregate counts, rates, and plan segments only — no emails, names, or user-id lists.
  • Events vs users are always reported distinctly (total occurrences vs unique distinct_id).
